# Практична робота 5

## Завдання 4
Зробити подвійне звільнення, яке інколи не падає через кешування малих блоків (tcache), але руйнує внутрішній стан allocator.

### Програмна реалізація
Для виконання завдання було створено програму, яка навмисно викликає помилку подвійного звільнення пам'яті.
* Програма виділяє невеликий блок пам'яті, який зазвичай потрапляє в кеш tcache для швидкого повторного використання;
* Ми двічі викликаємо free() для однієї адреси, що змушує систему помилково додати її до списку вільних блоків двічі.

### Результат виконання
При компіляції та запуску програма стикається з вбудованими засобами захисту системи:

**Попередження компілятора**
Компілятор gcc автоматично виявляє потенційну вразливість, оскільки ми намагаємося звільнити вже звільнений вказівник.
```
gcc -Wall task_4.c -o task_4
task_4.c: In function ‘main’:
task_4.c:10:5: warning: pointer ‘ptr’ used after ‘free’ [-Wuse-after-free]
   10 |     free(ptr);
      |     ^~~~~~~~~
```

**Вивід програми**
```
./task_4
Allocated at: 0x5df58a4a02a0
free(): double free detected in tcache 2
Aborted (core dumped)
```
Aborted це команда від glibc припинити роботу через виявлену спробу подвійного звільнення.

Хоча теоретично подвійне звільнення пам'яті могло б пошкодити внутрішні списки аллокатора, у сучасних системах Linux це призводить до негайного аварійного завершення програми з повідомленням. Це відбувається тому, що бібліотека glibc автоматично перевіряє цілісність кешу пам'яті (tcache) перед кожним викликом free(), блокуючи спробу повторного звільнення того самого блоку для забезпечення безпеки системи.
