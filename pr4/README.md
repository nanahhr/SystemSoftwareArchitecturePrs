## Завдання 4.1
Скільки пам'яті може виділити malloc(3) за один виклик?   
Максимальний обсяг пам’яті, який може бути виділений за один виклик malloc(3), дорівнює 2^64 Чому теоретично максимальний обсяг складає 8 ексабайт, а не 16?

### Програмна реалізація
Для виконання завдання було розроблено програму на мові C, яка досліджує максимальну межу виділення пам'яті за один запит.
* В якості аргументу використано константу SIZE_MAX, що відповідає максимальному значенню типу size_t (на 64-бітній платформі це 2^64 - 1);
* Програма перевіряє повернуте значення: якщо malloc не може зарезервувати такий блок, він повертає NULL, що й фіксується у виводі.

### Результат виконання
```
gcc task_4_1.c -o task_4_1
./task_4_1
```
**Вивід терміналу:**
```
./task_4_1
Trying to allocate: 18446744073709551615 bytes (2^64 - 1)
malloc: Cannot allocate memory
Result: malloc() returned NULL. Allocation failed.
```
Хоча тип size_t теоретично дозволяє адресувати до 16 ексабайт, результат тесту показав помилку "Cannot allocate memory".   

Також теоретичний максимум 8 ексабайт, а не 16 зумовлений аріхтектурними обмеженнями х86_64:
* Віртуальний адресний простір у 64-бітній Linux розділений навпіл;
* Нижня половина (8 ексабайт) виділена для процесів користувача;
* Верхня половина (8 ексабайт) зарезервована суворо для потреб ядра ОС.

## Завдання 4.2
Що станеться, якщо передати malloc(3) від’ємний аргумент? Напишіть тестовий випадок, який обчислює кількість виділених байтів за формулою num = xa * xb. Що буде, якщо num оголошене як цілочисельна змінна зі знаком, а результат множення призведе до переповнення? Як себе поведе malloc(3)?

### Програмна реалізація
Для виконання завдання було розроблено програму, яка демонструє поведінку malloc(3) при роботі з некоректними значеннями.
* Програма намагається виділити пам'ять, передавши malloc(-1). Оскільки malloc приймає аргумент типу size_t (беззнаковий тип), від'ємне число інтерпретується як максимально можливе значення;
* Також реалізовано кейс множення двох цілих чисел (xa * xb), що призводить до цілочисельного переповнення (integer overflow), внаслідок чого розмір алокації стає від'ємним.

### Результат виконання
```
gcc task_4_2.c -o task_4_2
./task_4_2
```

**Вивід програми**
```
./task_4_2
Attempting malloc(-1)...
malloc(-1) failed.

Calculating size: 1000000 * 1000000 = -727379968 (overflowed)
malloc(num) failed.
```
Отже: 
* malloc працює з типом size_t (беззнаковий), тому -1 перетворюється на величезне позитивне число (SIZE_MAX). Система не може виділити такий обсяг пам'яті, тому функція повертає NULL;
* Множення xa * xb призводить до того, що результат не вміщується у змінну int. Оскільки malloc не може виділити від'ємну кількість пам'яті, програма видає помилку алокації;
* Для уникнення помилок при обчисленні розмірів пам'яті завжди слід використовувати тип size_t, який виключає можливість появи від'ємних значень.

## Завдання 4.4
Чи є помилки у такому коді?
```
void *ptr = NULL;
while (<some-condition-is-true>) {
    if (!ptr)
        ptr = malloc(n);
    [... <використання 'ptr'> ...]
    free(ptr);
}
```
Напишіть тестовий випадок, який продемонструє проблему та правильний варіант коду.

### Програмна реалізація
Наведений фрагмент коду містить критичну логічну помилку: використання вказівника після його звільнення. Після виклику free(ptr) вказівник ptr продовжує зберігати стару адресу пам'яті. На наступній ітерації циклу умова if (!ptr) не спрацьовує, і програма намагається звернутися до пам'яті, яка вже була повернута.

### Результат виконання
При спробі компіляції програма генерує попередження, яке вказує на небезпечне використання пам'яті:
```
gcc -Wall task_4_4.c -o task_4_4
task_4_4.c: In function ‘main’:
task_4_4.c:16:9: warning: pointer ‘ptr’ used after ‘free’ [-Wuse-after-free]
   16 |         printf("Use-after-free at %p\n", ptr);
      |         ^~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
task_4_4.c:14:9: note: call to ‘free’ here
   14 |         free(ptr);
      |         ^~~~~~~~~

```
Після виправлення помилки програма працює коректно, не звертаючись до звільненої пам'яті.
```
./task_4_4
Used memory at 0x61e171bfc2a0
Used memory at 0x61e171bfc2a0

```
Було змінено:
* Вказівник ptr тепер оголошується всередині циклу for;
* Додано явне присвоєння ptr = NULL після free(ptr);
* Забрали умовну конструкцію if (!ptr), яка в початковому коді базувалася на застарілому значенні вказівника

## Завдання 4.5
Що станеться, якщо realloc(3) не зможе виділити пам’ять? Напишіть тестовий випадок, що демонструє цей сценарій.

### Програмна реалізація
Коли realloc не може виділити новий блок пам'яті, він повертає NULL, але старий блок пам'яті залишається незмінним. Якщо ми зробимо помилку і напишемо ptr = realloc(ptr, size) це призведе до витоку пам'яті.
### Результат виконання
```
gcc -Wall task_4_5.c -o task_4_5
./task_4_5
```
При спробі компіляції gcc видає попередження про перевищення розміру об'єкта, оскільки realloc не може фізично виділити SIZE_MAX байтів.
```
task_4_5.c:10:21: warning: argument 2 value ‘18446744073709551615’ exceeds maximum object size 9223372
036854775807 [-Walloc-size-larger-than=]                                                                 10 |     void *new_ptr = realloc(ptr, SIZE_MAX);
      |                     ^~~~~~~~~~~~~~~~~~~~~~
```
**Вивід програми:**
```
./task_4_5
Initial pointer: 0x5ea39ee802a0
realloc failed! Old pointer 0x5ea39ee802a0 is still valid.
```
Якщо система не може задовольнити запит, функція повертає NULL. Важливо, що при цьому початковий блок пам'яті залишається незмінним. Помилкове присвоєння ptr = realloc(ptr, size); без перевірки на NULL призведе до втрати початкового вказівника , що спричинить витік пам'яті.

## Завдання 4.6
Якщо realloc(3) викликати з NULL або розміром 0, що станеться? Напишіть тестовий випадок.

### Програмна реалізація
Для виконання завдання було розроблено програму, яка демонструє особливості realloc(3) при роботі з нестандартними аргументами.
* Згідно зі стандартом, realloc(NULL, size) еквівалентний malloc(size);
* Якщо передати 0 у якості розміру, realloc звільняє пам'ять (як free()) і повертає NULL або унікальний вказівник, який не можна використовувати для запису даних.
### Результат виконання
```
gcc task_4_6.c -o task_4_6
./task_4_6
```

**Вивід програми:**
```
./task_4_6
realloc(NULL, 16) allocated at: 0x601ba19782a0
realloc(ptr, 0) returned: (nil)

```
Отже можна побачити, що:
* realloc(NULL, size): Функція коректно виділяє пам'ять, поводячись як malloc(3);
* realloc(ptr, 0): Функція звільняє блок пам'яті, на який вказує ptr, і повертає NULL.
## Завдання 4.7

### Програмна реалізація
Для виконання завдання було розроблено програму, яка замінює стандартний realloc(3) на reallocarray(3).   

Основна перевага reallocarray полягає в тому, що він приймає кількість елементів та розмір одного елемента як окремі аргументи, що дозволяє системі внутрішньо перевіряти їх на множення, яке могло б призвести до переповнення цілих чисел.

reallocarray(void *ptr, size_t nmemb, size_t size) повертає NULL і встановлює errno в ENOMEM, якщо множення nmemb * size призводить до переповнення
### Результат виконання
Використання ltrace дозволяє побачити, як викликається функція:
```

```
Вивід ltrace:
```
ltrace ./task_4_7

reallocarray(0x5a40c4cc82a0, 500, 104, 0x19650)               = 0x5a40c4cc82a0
puts("Successfully reallocated to 500 "...Successfully reallocated to 500 elements.
)                   = 42
free(0x5a40c4cc82a0)                                          = <void>

```
reallocarray(3) є безпечнішою альтернативою realloc(3). Якщо добуток аргументів nmemb * size перевищує максимальний розмір size_t (integer overflow), функція автоматично поверне NULL і встановить код помилки ENOMEM.
