Для розуміння того, як програма розміщується у пам'яті, було використано утиліту size.

Отже:
* text (1839 байт): Є найбільшим, оскільки містить логіку виводу та стандартні бібліотечні виклики;

* bss (8 байт): Мінімальний, оскільки в програмі на даному етапі не оголошено великих глобальних масивів;
* Загальний розмір у десятковому форматі (dec) складає 2463 байти.

## Завдання 2.3
Скомпілюйте й запустіть тестову програму, щоб визначити приблизне розташування стека у вашій системі:
```
#include <stdio.h>
int main() {
    int i;
    printf("The stack top is near %p\n", &i);
    return 0;
}
```
Знайдіть розташування сегментів даних і тексту, а також купи всередині сегмента даних, оголосіть змінні, які будуть поміщені в ці сегменти, і виведіть їхні адреси.

Збільшіть розмір стека, викликавши функцію й оголосивши кілька великих локальних масивів. Яка зараз адреса вершини стека?
**Примітка**: стек може розташовуватися за різними адресами на різних архітектурах та різних ОС. Хоча ми говоримо про вершину стека, на більшості процесорів стек зростає вниз, до пам’яті з меншими значеннями адрес.

### Програмна реалізація
Спочатку було створено базову програму для перевірки адреси локальної змінної, яка за замовчуванням розміщується на вершині стека.
**Результат запуску**  
```
./task_2_3
The stack top is near 0x7ffd9be44b24
```

Потім код було розширено для виводу адрес усіх ключових сегментів. Оголошкно лобальні змінні (ініціалізовану та ні), виділено пам'ять у купі та виведено адресу самої функції main.
**Результат аналізу сегментів**  
```
./task_2_3
Text (code) segment: 0x61a046ff51a9
Data segment:        0x61a046ff8010
BSS segment:         0x61a046ff8018
Heap:                0x61a077d402a0
Stack top:           0x7fff7c939f3c
```
Результати підтверджують класичну теоретичну модель розподілу пам'яті:
* Адреси сегментів Text, Data та BSS знаходяться дуже близько одна до одної. Це "нижня" частина пам'яті, де зберігається сам код програми та глобальні змінні.
* Адреса купи (0x61a077d402a0) вища за статичні сегменти. Це область, яка виділяється під час роботи програми за допомогою malloc. Вона починається після сегмента BSS і росте "вгору" (у бік збільшення адрес).
* Адреса стека (0x7fff7c939f3c) відрізняється від інших. Вона знаходиться у "дуже високій" пам'яті (префікс 0x7fff).
* Дистанція між купою та стеком передбачена архітектурою ОС для того, щоб обидві області могли вільно рости назустріч одна одній без перекриття.

Далі було проведено експеримент для перевірки твердження, що стек росте "вниз" (від більших адрес до менших). Для цього код було модифіковано: додано функцію expand_stack, яка оголошує великий локальний масив розміром 1000 елементів int (приблизно 4000 байтів).
**Результат виконання**  
```
./task_2_3
Text (code) segment: 0x61563d6e9240
Data segment:        0x61563d6ec010
BSS segment:         0x61563d6ec02c
Heap:                0x61566ae552a0
Stack top:           0x7ffecac92fac
Stack level 1 (large array address): 0x7ffecac91fe0
Stack level 2 (large array address): 0x7ffecac91010
```
Порівняння адрес при заглибленні в рекурсію показало наступне:
* Вершина в main: ...2fac
* Рівень 1: ...1fe0 (адреса зменшилася)
* Рівень 2: ...1010 (адреса зменшилася ще більше)

Оскільки кожна наступна адреса на стеку є меншою за попередню, ми експериментально підтвердили, що в даній системі стек росте вниз (від старших адрес до молодших).

## Завдання 2.4
Дослідити механізм роботи стека процесу під час вкладених викликів функцій, використовуючи автоматизовані утиліти (gstack/pstack) та ручний аналіз через налагоджувач GDB.
* Створити код із ланцюжком викликів main() -> foo() -> bar() -> bar_is_now_closed(), де остання функція викликає блокуючий системний виклик pause().
* Підключитися до працюючого процесу (attach PID) та отримати трасування стека командою backtrace (bt).
* Використати утиліту gstack (або її аналоги) для отримання аналогічного результату та порівняти виводи.

### Програмна реалізація  
Для дослідження було написано програму, яка створює ланцюжок вкладених викликів: main() -> foo() -> bar() -> bar_is_now_closed(). Остання функція викликає системний виклик pause(), який блокує процес і дозволяє нам проаналізувати його стан.

**Результат виконання**  
При запуску програми ми бачимо адреси змінних у пам'яті. Адреси зменшуються з кожним новим викликом, що підтверджує ріст стека "вниз" (від старших адрес до молодших).
```
./task_2_4
My PID is: 2836
In function                 main; &localvar = 0x7ffd2c9c4064
In function                  foo; &localvar = 0x7ffd2c9c4034
In function                  bar; &localvar = 0x7ffd2c9c4014
In function    bar_is_now_closed; &localvar = 0x7ffd2c9c3ff4

 Now blocking on pause()...
```

Оскільки програма заблокована викликом pause(), використовуємо налагоджувач GDB у другому терміналі, щоб "приєднатися" до неї за її ідентифікатором PID:
```
sudo gdb -p 2836
```
Використовуємо команду bt, щоб побачити всі кадри стека, які зараз знаходяться в пам'яті. Це дозволяє побачити повний шлях, яким пройшла програма до моменту зупинки:

```
(gdb) bt
#0  0x000077130defa3d4 in __libc_pause () at ../sysdeps/unix/sysv/linux/pause.c:29
#1  0x00005c4bb6ec6244 in bar_is_now_closed ()
#2  0x00005c4bb6ec62a7 in bar ()
#3  0x00005c4bb6ec630a in foo ()
#4  0x00005c4bb6ec638f in main ()
```
Результат показує, що кадр #4 (main) є основою, а кадр #0 (pause) поточною точкою виконання.

Також було зроблено спробу використати утиліту gstack для автоматичного отримання трасування:
```
sudo gstack 2836
sudo: gstack: command not found
```
Оскільки утиліта відсутня в даній дистрибуції Ubuntu, аналіз через GDB є основним методом. Варто зауважити, що gstack технічно є лише оболонкою над GDB, тому отримані дані в обох випадках були б ідентичними.

Отже, при кожному виклику процедури в стеку створюється новий кадр (stack frame). Він зберігає адресу повернення та локальні змінні. Це дозволяє процесору точно знати, куди повернутися після завершення роботи поточної функції.

## Завдання 2.5
Відомо, що при виклику процедур і поверненні з них процесор використовує стек.Чи можна в такій схемі обійтися без лічильника команд (IP), використовуючи замість нього вершину стека? Обґрунтуйте свою відповідь та наведіть приклади.

### Відповідь
Ні, це неможливо. Попри те, що вони взаємодіють, вони мають різні функції: IP керує послідовністю дій, а стек зберігає дані.

IP автоматично переходить до наступної команди (інкремент). Стек працює за принципом LIFO (останній прийшов - перший пішов).
**Приклад**: У звичайному коді після команди №10 йде №11. IP просто додає одиницю. Якщо використовувати стек, то після команди №10 її адресу треба "викинути" (pop), щоб відкрити адресу №11. Але хтось мав заздалегідь покласти туди всі тисячі адрес вашої програми в зворотному порядку.

Єдиний момент, коли стек впливає на те, куди йти далі це повернення з функції.
**Приклад**: Коли підпрограма завершується інструкцією ret, процесор бере адресу зі стека і копіює її в IP. Стек лише підказує адресу, але виконує її все одно IP.

Отже, замінити лічильник команд (IP) вершиною стека неможливо, тому що це призведе до змішування керування програмою та обробки даних.

## Завдання варіанту 4
Дослідіть техніки розподілу пам’яті в багатопотокових програмах.

### Відповідь  
У багатопотокових системах стандартний менеджер пам'яті (malloc) використовує спільну область пам'яті. Коли декілька потоків одночасно намагаються виділити пам'ять, виникає потреба у блокуваннях. У результаті замість паралельної роботи потоки просто чекають один на одного, що значно знижує продуктивність на багатоядерних процесорах.
**Техніки оптимізації**  
* Розподіл пам'яті на декілька незалежних зон (арен). Кожен потік або група потоків працює зі своєю ареною, що дозволяє виділяти пам'ять паралельно без взаємних блокувань;
* Використання таких технік як TLAB (Thread-Local Allocation Buffers). Потік отримує великий блок пам'яті "наперед" і розподіляє його самостійно, не звертаючись до центрального менеджера.
* Заміна стандартної бібліотеки на рішення, оптимізовані для паралелізму: 
    - TCMalloc: фокусується на кешуванні малих об'єктів безпосередньо у потоках;
    - jemalloc: мінімізує фрагментацію та ефективно масштабується на велику кількість ядер.

Основним методом підвищення продуктивності в багатопотокових програмах є децентралізація розподілу пам'яті. Використання арен та локальних кешів дозволяє мінімізувати час очікування на синхронізацію та забезпечити масштабованість системи.

(gdb) bt
