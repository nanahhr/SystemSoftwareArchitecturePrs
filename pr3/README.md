## Завдання 3.1
Запустіть Docker-контейнер і поекспериментуйте з максимальним лімітом ресурсів відкритих файлів. Для цього виконайте команди у вказаному порядку.

### Програмна реалізація
Для автоматизації було використано Shell-скрипт task_3_1.sh.

**Виконання 1 (Звичайний користувач)**
```
./task_3_1.sh
1048576
```

Для звичайного користувача діє жорстке обмеження (Hard Limit). Користувач може вільно зменшувати свій ліміт, але будь-яка спроба підняти його вище поточної "стелі" блокується ядром. Коли користувач зменшив ліміт до 2000, він автоматично зробив 2000 своїм новим жорстким лімітом, тому не зміг повернутися до 3000.

**Виконання 2 (Користувач root)**
```
./task_3_1.sh
1048576
open files                          (-n) 1048576
open files                          (-n) 1048576
open files                          (-n) 3000
open files                          (-n) 3000
./task_3_1.sh: line 17: ulimit: open files: cannot modify limit: Operation not permitted
2000
open files                          (-n) 2000
open files                          (-n) 2000
./task_3_1.sh: line 28: ulimit: open files: cannot modify limit: Operation not permitted
```

Хоча всередині контейнера ми працюємо від імені root, результат виявився аналогічним. Це свідчить про те, що Docker-контейнер за замовчуванням запускається з обмеженими "можливостями".

Отримана помилка No permission свідчить про те, що Docker-контейнер має  обмеження на використання системних викликів профілювання (perf_event_open). Навіть маючи права root, процес не може отримати доступ до ресурсів ядра через встановлені політики безпеки хоста.

## Завдання 3.3
Напишіть програму, що імітує кидання шестигранного кубика. Імітуйте кидки, результати записуйте у файл, для якого попередньо встановлено обмеження на його максимальний розмір (max file size). Коректно обробіть ситуацію перевищення ліміту.

### Програмна реалізація
Для виконання завдання було розроблено програму на мові C, яка генерує випадкові результати кидків кубика та записує їх у текстовий файл.

* Використано системний виклик signal(SIGXFSZ, handle_sigxfsz) для перехоплення сигналу, який надсилає ядро ОС при спробі перевищити ліміт запису;
* Додано функцію fflush(), що забезпечує негайний запис даних з буфера на диск.

Перед запуском програми було встановлено обмеження на розмір файлу в 1 блок (1 КБ) за допомогою команди ulimit -f 1.
```
gcc -Wall task_3_3.c -o task_3_3
ulimit -f 1
```

### Результат виконання
```
./task_3_3
Simulating dice rolls and writing to a file...

The file size limit has been exceeded.
```
Результат виконання підтвердив, що встановлений ліміт ulimit -f 1 (1 КБ) блокує запис у файл на рівні системних викликів ядра, що призводить до негайної генерації сигналу SIGXFSZ.

## Завдання 3.4
Напишіть програму, що імітує лотерею, вибираючи 7 різних цілих чисел у діапазоні від 1 до 49 і ще 6 з 36. Встановіть обмеження на час ЦП (max CPU time) і генеруйте результати вибору чисел (7 із 49, 6 із 36). Обробіть ситуацію, коли ліміт ресурсу вичерпано.

### Програмна реалізація
Було розроблено програму на мові С, яка імітує нескінченний цикл розіграшів лотереї. Для перевірки роботи системних обмежень програма була запущена з попередньо встановленим лімітом на час використання процесора.

* Основний функціонал винесено в окрему функцію draw_lottery(), яка приймає параметри кількості чисел та діапазону;
* Для перехоплення сигналу від ОС використано функцію signal(SIGXCPU, handle_sigxcpu);
* У циклі while(1) додано математичне навантаження (операції з типом double), щоб програма швидше витрачала виділений ліміт процесорного часу.

### Результат виконання
Перед запуском було встановлено ліміт у 1 секунду процесорного часу за допомогою команди ulimit -t 1.
```
gcc task_3_4.c -o task_3_4
ulimit -t 1
./task_3_4
```
**Вивід програми:**
```
Starting endless lottery simulation to test CPU limit...
Drawing 7 from 49: 37 28 21 39 37 15 49 
Drawing 6 from 36: 22 29 6 18 18 15 
...
Drawing 7 from 49: 26 29 6 3 7 45 21 
Drawing 6 from 36: 20 22 16 9 8 36 
Killed
```
Результат виконання показав, що при досягненні ліміту в 1 секунду програма була примусово завершена операційною системою з повідомленням Killed.  

Це пояснюється тим, що при встановленні ліміту через ulimit -t система застосовує "жорстке" обмеження (Hard Limit). Оскільки програма створювала інтенсивне навантаження на ЦП, ядро ОС надіслало сигнал SIGKILL, який має найвищий пріоритет і не може бути перехоплений програмою

## Завдання 3.5
Напишіть програму для копіювання одного іменованого файлу в інший. Імена файлів передаються у вигляді аргументів.  
Програма має:
* перевіряти, чи передано два аргументи, інакше виводити "Program need two arguments";
* перевіряти доступність першого файлу для читання, інакше виводити "Cannot open file .... for reading";
* перевіряти доступність другого файлу для запису, інакше виводити "Cannot open file .... for writing";
* обробляти ситуацію перевищення обмеження на розмір файлу.

### Програмна реалізація
Було розроблено програму на мові С, що використовує низькорівневі системні виклики для маніпуляції файлами. Програма передбачає перевірку умов виконання та обробку сигналів ядра.

* Використано argc для контролю кількості вхідних параметрів;
* Використано перевірку результатів виклику open() для діагностики проблем із читанням чи записом;
* Встановлено обробник сигналу SIGXFSZ через функцію signal(), що дозволяє програмі реагувати на обмеження файлової системи ulimit -f.

### Результат виконання

```
# Тест 1: Відсутність аргументів
./task_3_5
# Вивід: Program need two arguments

# Тест 2: Відсутність вхідного файлу
./task_3_5 non_existent.txt dest.txt
# Вивід: Cannot open file non_existent.txt for reading

# Тест 3: Перевищення ліміту розміру файлу
ulimit -f 1
./task_3_5 source.txt dest.txt
# Вивід: File size limit exceeded (SIGXFSZ)

```

При встановленні ulimit -f 1 (1 КБ), системний виклик write() згенерував сигнал SIGXFSZ, як тільки обсяг записаних даних досяг межі. Обробник сигналу дозволив завершити роботу з інформативним повідомленням.

## Завдання 3.6
Напишіть програму, що демонструє використання обмеження (max stack segment size). Підказка: рекурсивна програма активно використовує стек.


### Програмна реалізація
Для було розроблено програму, яка використовує безкінечну рекурсію. Кожен виклик функції створює новий стековий кадр, що містить локальні змінні та адресу повернення, поступово заповнюючи виділений обсяг пам'яті.

* Функція recursive_function викликає саму себе, виділяючи локальний буфер розміром 64 КБ на кожній ітерації;
* У коді передбачено реєстрацію обробника сигналу SIGSEGV через системний виклик signal();
* Використано команду ulimit -s для встановлення штучного ліміту на розмір стека перед запуском процесу.

### Результат виконання
Було встановлено ліміт стека у 256 КБ, що при розмірі кадру у 64 КБ призводить до критичної помилки вже на початку виконання.

```
gcc task_3_6.c -o task_3_6

ulimit -s 256

./task_3_6
```

**Вивід програми**
```
Starting recursion with small stack limit...
Segmentation fault (core dumped)

```
Результат виконання показав виникнення помилки Segmentation fault, проте власне повідомлення програми про перехоплення сигналу не було виведено.

При вичерпанні стека пам'ять сегмента критично заповнюється і система не може виділити ресурси для створення нового стекового кадру під саму функцію-обробник сигналу handle_sigsegv. Оскільки для виклику будь-якої функції в С потрібен вільний простір у стеку, а його немає, ядро ОС миттєво припиняє процес без виконання програмного коду.

### Завдання 4
Реалізувати програму, що перевіряє максимальну кількість запущених процесів (ulimit -u).

### Програмна реалізація
Було розроблено програму на мові С, яка використовує системний виклик fork() у нескінченному циклі. Програма підраховує кількість успішно створених процесів, доки операційна система не відмовить у виділенні нового ідентифікатора (PID).

* Використано while(1) для безперервного виклику fork();
* Кожен дочірній процес виконує sleep(10), щоб залишатися активним у таблиці процесів і не звільняти місце завчасно;

### Результат виконання
Попри спробу встановити ліміт ulimit -u 50, система дозволила створити значно більшу кількість процесів.

```
gcc task_var_4.c -o task_var_4

ulimit -u 50

./task_var_4
```

**Вивід програми**
```
Starting process creation test (ulimit -u)...
Processes created: 10
...
Processes created: 4020
Processes created: 4030

Reached limit at 4036 processes.
Fork failed: Resource temporarily unavailable

```
* Перевірка через ulimit -a показала, що для користувача root у даному контейнері параметр max user processes встановлено як unlimited. Це пояснює, чому спроба обмежити кількість до 50 була ігнорована ядром на користь глобальних налаштувань контейнера;
* Зупинка на позначці 4036 є фактичним жорстким лімітом (Hard Limit) PIDs, встановленим для даного Docker-контейнера хост-системою;
* Отримання повідомлення Resource temporarily unavailable свідчить про те, що таблиця процесів повністю заповнена, і операційна система заблокувала створення нових об'єктів.

## Інструкція із запуску
### Підготовка системи
Для виконання роботи використовується ізольоване середовище Docker на базі образу Ubuntu. Також необхідно встановити інструменти для профілювання та моніторингу процесів.

У терміналі виконайте запуск контейнера та встановлення утиліт:
```
# Запуск Docker-контейнера
docker run -it --rm ubuntu bash

# Оновлення пакетів та встановлення компилятора, налагоджувача та системних утиліт
apt update
apt install -y build-essential gcc gdb git linux-tools-generic linux-tools-common
```

### Завантаження проекту
```
git clone https://github.com/nanahhr/SystemSoftwareArchitecturePrs.git
cd SystemSoftwareArchitecturePrs/pr3
```

### Компіляція та перевірка лімітів
Для кожного завдання розроблено відповідний вихідний код або скрипт. Загальна схема запуску виглядає наступним чином:
* Завдання 3.1 - 3.2: Використовуйте системні команди ulimit та perf stat безпосередньо в термінал;
* Завдання 3.3 - 3.6 та Варіант 4:
```
# Компіляція (на прикладі завдання 3.4)
gcc task_3_4.c -o task_3_4

# Встановлення ліміту (наприклад, ліміт часу CPU)
ulimit -t 1

# Запуск
./task_3_4
```

### Очищення системи
Після завершення всіх тестів достатньо вийти з термінала контейнера (команда exit). Оскільки контейнер було запущено з прапорцем --rm, усі тимчасові файли та встановлені пакети будуть автоматично видалені.
